## 実装指示書

### 1. 変更してよいファイル

`apps/random-picker/tests/e2e/e2e.spec.ts` だけを変更する。ほかのファイルは一切変更しない。`npm run test:e2e` は実行前に最新の `generated/index.html` を再生成するので、`generated/index.html` の手編集は禁止。

### 2. 既存ファイルの扱い

既存の `e2e.spec.ts` は、先頭の import、`appUrl`、`OpenedUrl` 型、`RandomPickerTestWindow` 型、`installBrowserApiStubs`、`getCopiedTexts`、`getOpenedUrls` は使い回してよい。
ただし、**既存の `test.describe(...)` は全部削除**してよい。
削除対象は、`初期表示`、`入力欄コピーボタンクリック`、`入力欄リンク起動ボタンクリック`、`パイプラインビルダー`、`出力欄コピーボタンクリック`、`出力欄リンク起動ボタンクリック` の中身すべて。

### 3. 最終的に残すテストタイトル

以下のタイトルを**完全一致**で作ること。
順番もこの順番に固定すること。

#### 初期表示

`test.describe("初期表示", () => {`

- `test("01 主要要素と初期状態を表示する", ...);`
  `});`

#### 入力欄

`test.describe("入力欄", () => {`

- `test("01 コピーボタン／入力内容をコピーできる", ...);`
- `test("02 オープンボタン／入力内容のURLを開ける", ...);`
  `});`

#### パイプライン

`test.describe("パイプライン", () => {`

- `test("01 追加・削除／ステップを追加して削除できる", ...);`
- `test("02 並び替え／ステップ順の変更を実行結果に反映できる", ...);`
- `test("03 実行／追加の入力欄がなく、出力欄の内容を使わない処理を実行できる", ...);`
- `test("04 実行／追加の入力欄がある処理を実行できる", ...);`
- `test("05 実行／出力欄の内容を使う処理を実行できる", ...);`
  `});`

#### 出力欄

`test.describe("出力欄", () => {`

- `test("01 コピーボタン／出力内容をコピーできる", ...);`
- `test("02 オープンボタン／出力内容のURLを開ける", ...);`
  `});`

このタイトル構成にする理由は、E2E が見るべき主要ユーザー操作を DOM 順に並べつつ、パイプライン実行の3件は「どの processor 名か」ではなく「追加入力欄の有無」「出力欄内容の利用有無」という抽出基準で分けるため。`processor-registry.ts` でも、`ランダム抽出` だけ `抽出件数` を持ち、`前回出力を除外` だけ前回出力を使う。

### 4. 共通実装ルール

各テストは必要なものだけセットアップすること。
copy / open テストだけ `installBrowserApiStubs(page)` を呼ぶ。
初期表示とパイプライン系テストでは不要なので呼ばない。
各テストの先頭で `await page.goto(appUrl);` を呼ぶこと。
Playwright の locator は、できるだけ既存 DOM の `id` と `aria-label` を使うこと。具体的には `#input`、`#output`、`#processorSelect`、`#addStepBtn`、`#pipelineRunBtn`、`button[name]` 相当の `getByRole("button", { name: ... })` を使う。

テスト名に意味のない具体値を書くな。
ただし、**テスト本文の入力値や期待値は具体的に固定**すること。
タイトルは抽出基準を書く。本文は代表例を具体値で書く。
この2つを混同しないこと。

#### 入力欄 / 02 オープンボタン／入力内容のURLを開ける

**8パターン展開は書かない。**

#### パイプライン / 01 追加・削除／ステップを追加して削除できる

このテストでは次の手順に固定する。

1. `page.goto(appUrl)` を呼ぶ
2. `#processorSelect` で `"trim"` を選ぶ
3. `#addStepBtn` をクリックする
4. `.pipeline-step-item` が1件になることを確認する
5. `.pipeline-step-name` が `"空白削除"` になることを確認する
6. `#processorSelect` で `"filterEmpty"` を選ぶ
7. `#addStepBtn` をクリックする
8. `.pipeline-step-item` が2件になることを確認する
9. `.pipeline-step-name` が `["空白削除", "空行除外"]` になることを確認する
10. 2件目の `.pipeline-step-item` の中にある `.pipeline-delete-btn` をクリックする
11. `.pipeline-step-item` が1件になることを確認する
12. `.pipeline-step-name` が `["空白削除"]` になることを確認する

このテストでは **実行しない**。
目的は追加と削除だけ。

#### パイプライン / 02 並び替え／ステップ順の変更を実行結果に反映できる

このテストでは次の手順に固定する。

1. `page.goto(appUrl)` を呼ぶ
2. `#processorSelect` で `"trim"` を選ぶ
3. `#addStepBtn` をクリックする
4. `#processorSelect` で `"filterEmpty"` を選ぶ
5. `#addStepBtn` をクリックする
6. `.pipeline-step-name` が `["空白削除", "空行除外"]` であることを確認する
7. `const dataTransfer = await page.evaluateHandle(() => new DataTransfer());` を作る
8. 1件目の `.pipeline-step-item` に対して `dragstart` を dispatch する
9. 2件目の `.pipeline-step-item` に対して `dragover` を dispatch する
10. 並び替え後の `.pipeline-step-name` が `["空行除外", "空白削除"]` になることを確認する
11. 並び替え後の 2件目の `.pipeline-step-item` に対して `dragend` を dispatch する
12. `#input` に `" \n A "` を入力する
13. `#pipelineRunBtn` をクリックする
14. `#output` の値が `"\nA"` であることを確認する

このケースで使う processor の組み合わせは固定する。
理由は、`空白削除 → 空行除外` なら `"A"` になるが、`空行除外 → 空白削除` なら `"\nA"` になり、順序差が明確に出るため。
**`dragTo` を使うかどうかで迷わないこと。上記の `dispatchEvent` 方式で固定すること。**

#### パイプライン / 03 実行／追加の入力欄がなく、出力欄の内容を使わない処理を実行できる

このテストでは次の手順に固定する。

1. `page.goto(appUrl)` を呼ぶ
2. `#processorSelect` で `"trim"` を選ぶ
3. `#addStepBtn` をクリックする
4. `#processorSelect` で `"filterEmpty"` を選ぶ
5. `#addStepBtn` をクリックする
6. `#input` に `"  A  \n\n  B  \n  C  "` を入力する
7. `#pipelineRunBtn` をクリックする
8. `#output` の値が `"A\nB\nC"` であることを確認する

このケースは、**追加の入力欄が出ない**、**出力欄の既存内容を使わない** 代表ケースとして固定する。
タイトルに processor 名は書かないが、本文では代表例として `trim` と `filterEmpty` を使ってよい。

#### パイプライン / 04 実行／追加の入力欄がある処理を実行できる

このテストでは次の手順に固定する。

1. `page.goto(appUrl)` を呼ぶ
2. `#processorSelect` で `"pickRandom"` を選ぶ
3. `#addStepBtn` をクリックする
4. `.pipeline-step-item` が1件であることを確認する
5. その `.pipeline-step-item` 内に `input[type="number"]` が1件存在することを確認する
6. その入力欄の値が初期値 `"1"` であることを確認する
7. その入力欄を `"2"` で上書きする
8. `#input` に `"A\nB\nC\nD"` を入力する
9. `#pipelineRunBtn` をクリックする
10. `#output` の値を改行で分割し、空行除外した配列を `resultLines` とする
11. `resultLines.length === 2` を確認する
12. `resultLines` の各要素が `["A", "B", "C", "D"]` のいずれかであることを確認する
13. `new Set(resultLines).size === 2` を確認する

このケースでは、**出力の完全一致は期待しない**。
乱択なので、判定基準は「2件出る」「元の入力集合に含まれる」「重複しない」の3つで固定する。

#### パイプライン / 05 実行／出力欄の内容を使う処理を実行できる

このテストでは次の手順に固定する。

1. `page.goto(appUrl)` を呼ぶ
2. `#input` に `"A\nB\nC"` を入力する
3. ステップを1件も追加しない
4. `#pipelineRunBtn` をクリックする
5. `#output` の値が `"A\nB\nC"` であることを確認する
6. `#processorSelect` で `"excludePrevious"` を選ぶ
7. `#addStepBtn` をクリックする
8. `#input` に `"A\nB\nC\nD"` を入力する
9. `#pipelineRunBtn` をクリックする
10. `#output` の値が `"D"` であることを確認する

このケースは、`前回出力を除外` が **出力欄の前回結果を使っている** ことを、ユーザー操作だけで確認する代表ケース。
出力欄へ `evaluate()` で直接値を入れてはいけない。最初の実行で前回出力を作ること。

### 6. 書き方の強制ルール

`入力欄コピーボタンクリック` のような旧タイトルは使わない。
タイトルは今回指定したものに完全一致させること。
`// パターン整理` `// パターン一覧` の大きな総当たりコメントは全部削除すること。
新しく入れるコメントは最小限でよいが、日本語で書くこと。
