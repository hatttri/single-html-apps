# `ui.pipelineRunBtn.onclick` インテグレーションテスト作り直し指示書

## 目的

`apps/random-picker/tests/integration/event.test.ts` 内の `describe("ui.pipelineRunBtn.onclick", ...)` を、
**主条件が一目で分かる名前** に統一して、**既存ケースを全削除して1から作り直す**。

この作業は **`ui.pipelineRunBtn.onclick` の describe ブロックだけ** を対象にする。
他の `describe` は触らない。

---

## この作業の大前提

1. **既存の `describe("ui.pipelineRunBtn.onclick", ...)` の中身は全部削除すること。**
   - 既存ケースの流用
   - 既存ケースの文言修正
   - 既存ケースの一部だけ残す
   - 既存ケースの順番入れ替えだけで済ませる
     これらはすべて禁止。**完全に新しく書き直すこと。**

2. この作業は **インテグレーションテストの整理だけ** を行う。
   - `src/event.ts`
   - `src/pipeline.ts`
   - `src/processor-registry.ts`
   - `src/processor.ts`
   - `src/dom.ts`
   - `src/script.ts`
     などの**本番コードは変更禁止**。

3. `event.test.ts` 全体の既存方針は維持すること。
   - `initApp` は使わない
   - `createUi()` と `bindInputEvent` / `bindPipelineEvent` / `bindOutputEvent` を使う
   - DOM event を発火して連携を見る
   - ブラウザ外部副作用以外はモックしない
     これは既存の `event.test.ts` 再構築指示書と同じ方針である。

4. `AGENTS.md` に従い、インテグレーションテストは `describe("targetElement.onEvent", ...)` 配下に `正常系` / `異常系` を置く構造にすること。

---

## 今回の確認対象

`ui.pipelineRunBtn.onclick` で主として確認したい条件は次の 5 つだけである。

### 正常系

- ステップ0件
- ステップ1件
- ステップ2件以上
- 出力あり
- 2回連続実行

### 異常系

- 未登録ステップを含む
- 不正なパラメータ値

この 7 件に絞ること。
**それ以外のケースを増やしてはいけない。**

---

## テスト名の強制ルール

### 1. テスト名は必ずこの文言にする

```ts
describe("ui.pipelineRunBtn.onclick", () => {
  describe("正常系", () => {
    test("ステップ0件", () => {});
    test("ステップ1件", () => {});
    test("ステップ2件以上", () => {});
    test("出力あり", () => {});
    test("2回連続実行", () => {});
  });

  describe("異常系", () => {
    test("未登録ステップを含む", () => {});
    test("不正なパラメータ値", () => {});
  });
});
```

### 2. 次のような名前は禁止

- `steps.length=0 / ...`
- `trim 1件 / ...`
- `trim -> filterEmpty / ...`
- `excludePrevious 1件 / ...`
- `1回目実行後に input.value を変更 / ...`
- `1回目実行後の output.value がある / ...`

理由:

- インテグレーションテストで内部寄りすぎる
- 主条件より実装手段が前に出る
- 名前を見ただけで整理軸が分かりにくい
  既存ケースはまさにこの状態なので、今回はそれをやめる。

### 3. テスト名に書いてよいのは主条件だけ

- `ステップ0件`
- `ステップ1件`
- `ステップ2件以上`
- `出力あり`
- `2回連続実行`
- `未登録ステップを含む`
- `不正なパラメータ値`

これ以外の条件は、テスト本文の中で**代表的な正常値**を使って実現すること。
名前に細かく書かないこと。
この「主条件だけを書く」方針は、最近の指示書でも明示されている。

## 各ケースで何を確認するか

## 1. 正常系 `ステップ0件`

### 目的

パイプラインが空でも、実行ボタン押下で入力内容がそのまま出力欄に入ることを確認する。

### セットアップ条件

- `appState.pipeline` は空のまま
- `ui.input.value` に代表的な複数行文字列を入れてよい
- `ui.output.value` は空でよい

### 主な確認

- `ui.output.value` が入力内容と一致すること

### やってはいけないこと

- ステップ追加をしない
- `processorSelect` を使わない
- `renderPipelineStepList` を個別に触ってケースを複雑にしない

---

## 2. 正常系 `ステップ1件`

### 目的

ステップが1件ある状態で、実行結果が出力欄に反映されることを確認する。

### セットアップ条件

- 追加するステップは **1件だけ**
- 具体的にどのステップを使うかはテスト名に書かない
- 本文では代表例として `trim` を使ってよい

### 主な確認

- 1件のステップが適用された結果が `ui.output.value` に入ること

### やってはいけないこと

- 2件以上のステップを入れない
- `出力あり` や `2回連続実行` の条件を混ぜない

---

## 3. 正常系 `ステップ2件以上`

### 目的

ステップが2件以上ある状態で、順に実行された結果が出力欄に反映されることを確認する。

### セットアップ条件

- ステップは **2件以上**
- 最小で 2 件にする
- 本文では代表例として 2 件使えばよい
- テスト名に具体的なステップ名は書かない

### 主な確認

- 複数ステップ適用後の結果が `ui.output.value` に入ること

### やってはいけないこと

- 3件以上に増やして無駄に複雑にしない
- `出力あり` 条件を混ぜない
- `2回連続実行` にしない

---

## 4. 正常系 `出力あり`

### 目的

実行前に出力欄に値がある場合、その値が `previousOutput` として扱われる連携を確認する。

現在の `handlePipelineRunBtnClick` は、クリック時に `previousOutput: deps.ui.output.value ?? ""` をコンテキストに入れて `executePipeline` に渡している。したがって、このケースは「出力あり」が主条件になる。

### セットアップ条件

- `ui.output.value` に事前値を入れる
- その事前値を意味のある形で参照するステップ構成にする
- 本文では代表例として `excludePrevious` を使ってよい
- テスト名に `excludePrevious` とは書かない

### 主な確認

- 実行前の `ui.output.value` が考慮された結果になっていること

### やってはいけないこと

- 2回実行しない
- `2回連続実行` のケースと兼用しない

---

## 5. 正常系 `2回連続実行`

### 目的

連続実行時に、1回目の結果を踏まえて2回目が動くことを確認する。

### セットアップ条件

- 同じテスト内で `ui.pipelineRunBtn.click()` を **2回** 呼ぶ
- 1回目実行後に必要な値変更を行ってよい
- 2回目の入力値や既存出力値が、最新状態として使われるような代表例を作る
- ただしテスト名にはそれを書かない

### 主な確認

- 2回目実行後の `ui.output.value` が期待値になること

### 重要

このケースでは、従来の

- `1回目実行後に input.value を変更 / 2回目は最新 input.value を使う`
- `1回目実行後の output.value がある / 2回目は最新 output.value を previousOutput として使う`

のような**細分化された観点を1本に統合する**。
今回の整理軸では、主条件はあくまで **2回連続実行** だからである。

---

## 6. 異常系 `未登録ステップを含む`

### 目的

未登録のステップ ID がパイプラインに含まれていても、既知のステップだけで処理が継続されることを確認する。

`executePipeline` は、ステップに対応する定義が見つからない場合 `continue` でそのステップを飛ばす構造になっているため、この異常系は `ui.pipelineRunBtn.onclick` でも意味がある。既存ケースにもこの観点はあるが、現在は `input.value が1行` と `input.value が複数行` に分裂しており、主条件がぼやけている。今回は1件に統合する。

### セットアップ条件

- `appState.pipeline` を直接差し替えてよい
- 未登録 ID と登録済み ID を混在させる
- ケースは **1本だけ**

### 主な確認

- 未登録ステップがあっても、既知のステップの結果として `ui.output.value` が期待どおりになること

### やってはいけないこと

- `input.value が1行`
- `input.value が複数行`
  のように派生ケースへ分けない

---

## 7. 異常系 `不正なパラメータ値`

### 目的

不正なパラメータ値が渡されたときの挙動を確認する。

### セットアップ条件

- 不正値を必要とするステップを 1 件用意する
- 本文では代表例として `pickRandom` の件数に不正値を入れてよい
- テスト名に `pickRandom` とは書かない

### 主な確認

- 現在の本番挙動に合わせて確認すること
- 例外が投げられるなら `toThrow`
- 例外が投げられず結果が出るならその結果
  のように、**現行実装どおり** に書くこと

### 重要

このケースは**仕様を決めるためのテストではない**。
**現在の実装が実際にどう動くか** を確認して、それに合わせて書くこと。
テストを書くために本番コードを変えてはいけない。
